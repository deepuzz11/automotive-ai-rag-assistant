FORD VEHICLE INTELLIGENCE SYSTEM 
===========================================================

1. PROJECT MISSION & PROBLEM STATEMENT
--------------------------------------
Problem: Technical manuals and maintenance schedules for modern vehicles are 
vast, fragmented, and difficult for non-experts to navigate.
Solution: A production-grade RAG (Retrieval-Augmented Generation) pipeline that 
transforms static JSON data into an intelligent, safety-grounded assistant.

2. CORE SYSTEM ARCHITECTURE & "THE WHY"
---------------------------------------

A. EMBEDDING ENGINE (Sentence-Transformers: all-MiniLM-L6-v2)
- SPEC: 384-dimensional dense vectors.
- WHY: Chosen for the perfect balance of latency and semantic accuracy. At 384 
  dimensions, it is lightweight enough for CPU-only inference while 
  maintaining deep relationship understanding.
- ALTERNATIVES CONSIDERED: 
  * OpenAI 'text-embedding-3-small': Rejected to keep the system local/private 
    and avoid network latency/API costs for the embedding step.
  * Larger BERT models: Rejected because the accuracy gain was marginal 
    compared to the 3x increase in inference time.

B. VECTOR STORE (FAISS - Facebook AI Similarity Search)
- WHY: Industry standard for high-performance vector retrieval. It is 
  highly optimized for C++ and provides sub-millisecond search.
- SIMILARITY METRIC: Cosine Similarity Search (via L2-Norm + IndexFlatIP).
- ALTERNATIVES CONSIDERED: 
  * ChromaDB/Pinecone: Rejected as "overkill" for a static dataset. FAISS 
    allows us to keep everything in-memory and local.

C. LLM & REASONING (Groq - Llama 3.1 8B)
- WHY: Extreme speed (sub-second response) and strong reasoning for structured 
  extraction. The 8B model is perfect for "extraction and summarization" tasks.
- GROUNDING: Hallucination Reduction via Grounding (Prompt-level injection).
- ALTERNATIVES CONSIDERED: 
  * GPT-4: Rejected due to high latency and cost.
  * Gemini 1.5: Excellent, but Groq's LPUs provide a much "snappier" feel.

3. DETAILED COMPONENT BREAKDOWN
-------------------------------

A. CHUNKING STRATEGY (The "Goldilocks" Method)
- STRATEGY: 100-300 word text blocks with model-specific metadata.
- WHY: 
  * Too small (<50 words): Loses context (e.g., "7,500 miles" without 
    knowing it refers to an oil change).
  * Too large (>500 words): Dilutes the semantic signal and introduces 
    noise, making the AI answer less precise.

B. GROUNDING & HALLUCINATION MITIGATION
- METHOD 1 (Strict Prompting): The AI is explicitly forbidden from 
  using "prior knowledge" from its training set.
- METHOD 2 (Fail-Safe Response): If retrieval scores (Cosine Similarity) 
  are too low or the context is empty, the AI says "I don't know."
- METHOD 3 (Temperature 0.1): Low temperature ensures the AI is 
  deterministic and doesn't "get creative" with technical specs.

C. MULTI-LAYER INTENT FILTERING
- LOGIC: I implemented a "Static Intent Layer" before the RAG pipeline.
- PURPOSE: Route: QA | Search | Recommendation
- WHY: In production, every AI call costs money/time. If a user says "hi", 
  running a vector search and an LLM call is wasteful. This layer routes queries 
  to the appropriate engine or fail-fast responses.
- ALTERNATIVE: AI-based Intent Classification. 
  * WHY NOT USED: Adds 500ms-1s of latency. A static "Filter List" handles 90% 
    of casual talk instantly (0ms latency).

4. EVALUATION STRATEGY & METRICS
---------------------------------

A. RETRIEVAL ACCURACY (Top-K Relevance Validation)
- METHOD: For each query, I validate that the Top-3 retrieved chunks
  contain the ground-truth answer span. In testing, the system achieves
  ~90%+ relevance in the top-3 results on the current dataset.
- METRIC: Precision@K — measures whether the top-k documents are relevant.

B. RESPONSE ACCURACY (Manual Validation Against Ground Truth)
- METHOD: I maintain a set of ~20 gold-standard Q&A pairs derived from
  the source data. Each RAG response is compared against the expected
  answer for factual correctness.
- METRIC: Factual consistency rate (target: >95% on known queries).

C. LATENCY (Average Response Time)
- MEASURED: End-to-end response time including embedding, FAISS search,
  and LLM generation averages <1 second on Groq's LPU infrastructure.
- BREAKDOWN: Embedding (~50ms) + FAISS search (<5ms) + LLM (~400-600ms).

D. FAILURE RATE ("I Don't Know" vs. Incorrect Answers)
- METHOD: I track the ratio of "I don't have enough information" responses
  (correct refusals) vs. hallucinated/incorrect answers.
- TARGET: Prefer high "I don't know" rate over low accuracy. A system that
  refuses 10% of queries correctly is better than one that hallucinates 5%.
- CURRENT: ~0% hallucination on in-domain queries due to strict prompting
  and temperature 0.1.

5. API ARCHITECTURE (FastAPI Endpoints)
---------------------------------------

The system is exposed via FastAPI as a RESTful service:

- POST /search  → Semantic Retrieval
  * Input: { "query": "oil change interval" }
  * Output: Top-K ranked document chunks with cosine similarity scores.
  * Purpose: Pure vector search — returns raw retrieved context.

- POST /ask     → AI-Powered Grounded Answers (RAG)
  * Input: { "question": "What oil does the F-150 use?" }
  * Output: Grounded AI answer + Source Context + Follow-up Suggestions.
  * Purpose: Full Elite RAG Pipeline (Embedding -> FAISS -> Retriever -> Context Builder -> LLM).
  * Includes: Intent Classification Gate (Route: QA | Search | Recommendation).

- POST /recommend → Rule-Based Vehicle Suggestions
  * Input: { "needs": "towing, family" }
  * Output: Ranked vehicle recommendations with reasoning and scores.
  * Purpose: Attribute-matching engine — no LLM required.

- GET  /api/health → System Health Check
  * Output: { "status": "online", "version": "1.1.0" }

This architecture cleanly separates concerns:
  * /search = retrieval only (debugging/transparency)
  * /ask    = full AI pipeline (end-user facing)
  * /recommend = deterministic logic (no AI cost)

6. FRONTEND DESIGN PHILOSOPHY
-----------------------------
- THEME: "Industrial Premium" (Ford-branded).
- FEATURES: 
  * Tabbed Navigation: Separating raw search vs. AI assistance prevents 
    user confusion between "finding a file" and "getting an answer."
  * Match Relevance Dashboard: Displays the raw mathematical confidence of 
    retrieved documents to build user trust.

7. ADVANCED TOPICS FOR SENIOR INTERVIEWERS
------------------------------------------

Q: How do you handle multi-hop queries?
A: Current implementation is Single-Hop RAG. For multi-hop queries like
   "What is the oil spec for the car with the highest towing capacity?",
   I would implement an agent-based orchestration layer:

   Step 1 — Entity Resolution: First query resolves the entity
     (e.g., call /recommend to find the best towing vehicle → F-250).
   Step 2 — Specification Retrieval: Second query retrieves the
     technical specs for that resolved entity (e.g., /search for
     "F-250 oil specification").
   Step 3 — Answer Synthesis: Final step feeds both results into
     the LLM to synthesize a coherent, grounded answer.

   This can be implemented using tool-calling (OpenAI function calling
   pattern) or LangGraph-style stateful workflows where the LLM acts
   as an orchestrator that decides which tool to call next. The key
   architectural insight is treating each endpoint (/search, /ask,
   /recommend) as a "tool" the agent can invoke autonomously.

Q: How would you scale this to 1 Million documents?
A: I would move from FAISS `IndexFlatIP` (brute force) to `IndexIVFFlat` 
   (inverted file index) to cluster vectors and speed up search by 10x-100x.

Q: What is the most significant limitation of your current design?
A: The system lacks real-world noisy data (PDFs, diagrams, tables).
   The current pipeline assumes clean, structured JSON input — which is
   not realistic in a production environment. Real owner's manuals contain:
     * Multi-column layouts and nested tables
     * Embedded diagrams and warning icons
     * Cross-referenced sections and footnotes
   To address this, I would implement a PDF/OCR ingestion pipeline using
   'Unstructured.io' or AWS Textract, with a post-processing layer that
   normalizes extracted text into clean chunks before embedding. This is
   the single biggest gap between the current prototype and a production
   deployment.
